# redis-commands

The command layer: the dispatch router plus every command implementation. Top
of the library stack — depends on `redis-core`, `redis-ds`, `redis-protocol`,
and `redis-types`.

## Owns (canonical types — `harness/type-vocabulary.tsv`)
`CommandSpec` — `src/spec.rs` (audit mode).

## Module map (28 files)
- router            dispatch (the command table + `dispatch(ctx)` entry point)
- generated table   generated.rs (425 commands; build output — see below)
- data-type impls   string, list, set, zset, hash, stream, bitops, geo
                    (+ geohash helpers), hyperloglog, sort
- server commands   connection, info, slowlog_cmd, multi, persist, pubsub,
                    replication, replica_dialer, aof, eval (scripting)
- module-compat     bloom, json, vector — Rust-native subsets of
                    RedisBloom/JSON/vector, NOT C-module ABI ports

## generated.rs — do not hand-edit
`src/generated.rs` is the command metadata table (arity, flags, key specs),
auto-generated by `harness/gen-command-registry.py` from
`reference/valkey/src/commands/*.json`. Regenerate after pulling upstream:
`python3 harness/gen-command-registry.py`. Never edit it by hand.

## Does NOT own
- server/db/object/client state  → `redis-core`
- wire encode/decode             → `redis-protocol`
- value encodings                → `redis-ds`

## Footguns
- Two tables, two jobs: `generated.rs` holds *metadata* (from JSON); `dispatch.rs`
  holds the *handler wiring* (`DispatchEntry { name, handler }`). A command needs
  both — metadata alone won't run, a handler alone won't validate arity.
- `dispatch.rs` has layered lookup (hot / indexed / runtime) behind `OnceLock`s.
  Those lazy-init on first traffic — a known cold-start latency tail; eager-init
  is the fix, don't add more lazy statics on the hot path.
- `bloom`/`json`/`vector` are *our* Rust reimplementations; they will not match
  the C module test suites and aren't meant to.

## Common tasks / where to look
- add a command handler → write `fn <x>_command(ctx: &mut CommandContext) ->
  RedisResult<()>` in the matching type module, then add
  `DispatchEntry { name: b"X", handler: crate::<mod>::x_command }` to the table
  in dispatch.rs.
- new upstream command (metadata) → add the JSON under `reference/`, rerun
  `gen-command-registry.py`. Don't touch generated.rs directly.
- find an existing command's impl → grep dispatch.rs for `b"NAME"` → handler path.
