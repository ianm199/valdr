Valdr

Valkey, in safe Rust.

Wire-compatible single-node alpha. Memory-safe TLS via rustls — no OpenSSL, no fork-based persistence, no loadable C module ABI.

Quick start

docker pull ghcr.io/flightdecksystems/valdr:alpha
docker run -d --name valdr -p 6379:6379 ghcr.io/flightdecksystems/valdr:alpha
printf 'PING\r\n' | nc 127.0.0.1 6379
# +PONG

Or build from source: cargo build --release --bin redis-server.

What works

Strings, hashes, lists, sets, sorted sets. Streams with consumer groups (XADD, XREAD, XREADGROUP, XACK, XCLAIM). Pub/Sub, sharded and regular. Scripting (EVAL, EVALSHA, FCALL, FUNCTION LOAD). Transactions (MULTI, EXEC, WATCH). RDB persistence in both directions — we save and the C Valkey binary loads, C Valkey saves and we load. AUTH and full ACL. TLS via rustls. INFO, CLIENT INFO, CLIENT LIST, COMMAND, SLOWLOG, and CONFIG GET / SET for the supported keys.

What doesn't yet

Replication code is in the tree (REPLICAOF, PSYNC, REPLCONF, WAIT) but the integration test suite has never been run against Valdr — partial resync in particular is unproven. Don't depend on replication. AOF persistence is alpha; the rewrite path has known gaps and is not release-gated. Cluster mode and Sentinel are not implemented. Loadable C modules (the .so Redis Module API) are intentionally not — Rust-native subsets of RedisJSON, RedisBloom, and a vector type exist instead.

A handful of unit/tls.tcl tests assert OpenSSL CBC ciphers (AES256-SHA256, AES128-SHA256). rustls refuses CBC ciphers on principle; those tests cannot pass without abandoning the safety thesis. That is a security upgrade, not a regression, but it is a deliberate divergence from upstream Valkey worth knowing about.

Why this exists

The security thesis.

The high-severity Redis CVEs of the past decade have been memory-safety bugs in C. CVE-2025-49844 (RediShell), CVSS 10.0, was a use-after-free in the Lua garbage collector that lived in the source for thirteen years. CVE-2025-46817 was an integer overflow in unpack. CVE-2021-32625 was heap corruption through STRALGO LCS. The pattern is consistent: the bug is in the substrate, not the protocol.

Valdr replaces the substrate. The data engine, the RESP parser, and the TLS stack are all safe Rust. There is no fork(). There is no loadable C module ABI. The honest exception: Valdr currently embeds the Lua VM via mlua, which wraps the same upstream C interpreter. The Lua subsystem is not yet memory-safe Rust — see MLUA_EXIT_PLAN.md for the path to replacing it with the sibling lua-rs runtime. Until that lands, the RediShell class of vulnerability is not eliminated by Valdr; everything outside the Lua VM is.

The methodology.

Valdr was built with a custom AI-driven C-to-Rust porting harness: upstream Valkey pinned as source, the upstream test suite as oracle, a type-vocabulary registry that enforces single-owner cross-crate types, mechanical Stop hooks gating every commit (per-crate unsafe budget, forbidden patterns, PORT STATUS trailers on every .rs file), and bounded subagent roles whose write access is structurally constrained. The Rust ports are how we prove the harness works. Next: nginx.

Try it

Minimal TLS-enabled config — a six-line redis.conf: 

bind 0.0.0.0
port 0
tls-port 6379
tls-cert-file /certs/server.crt
tls-key-file  /certs/server.key
requirepass   <a long random string>

Connect: valkey-cli --tls --cacert ./server.crt -h <host> -p 6379 -a '<password>' PING.
