The tuning trajectory

Performance work is tied to a reproducible objective: the profile matrix is the harness's optimization target, rerun after each focused packet. Deep-pipeline GET moved from ~221k req/s to ~2.17M req/s — roughly a 10× improvement — across a sequence of hot-path passes:

# each line is a landed tuning packet, oldest first
batch replies on the write side
drain the query buffer once per read batch
direct-write ordinary request/reply traffic
batch client-info snapshots
reuse argv storage instead of reallocating
monotonic timing on the hot path
hold the DB0 lock across safe read batches
cache generated command metadata
skip argv snapshots unless slowlog / AOF / replication need them
skip idle standalone replication propagation
fold handler + metadata lookup into one dispatch table
bucket the dispatch table by command prefix
Every benchmarked commit is recorded and plotted on a live, auto-built dashboard — each point keyed to its commit and hardware fingerprint so results from different machines are never silently merged.

The gap that remains is architectural

This is stated plainly because it is the honest read. valkey-rs is still a blocking thread-per-connection runtime that shares DB state through Arc<Mutex<RedisDb>>. Upstream Valkey's advantage on tiny pipelined commands comes from single-threaded event-loop ownership and batching with no locking.

Closing it is a known piece of work, not a mystery: the next performance milestone — moving to an event-loop ownership model — is specified in RUNTIME_OWNERSHIP_PLAN.md. This is not a claim of being "faster than C"; it is a memory-safe reimplementation that is already competitive — within a small constant factor on average, and ahead on real-work commands.

Reproduce it

# standard suite: -n 1,000,000 -c 50 -P 100 -d 64
$ bash harness/bench/run.sh

# the profile matrix (the harness optimization target)
$ bash harness/bench/run-profile-matrix.sh

# rebuild the live history dashboard from recorded results
$ python3 harness/bench/history.py
