The tuning trajectory

Performance work is tied to reproducible objectives: default-suite parts, pipeline smoke, JSON document mix, and profiler probes. Deep-pipeline GET moved from ~221k req/s to ~4.55M req/s in the latest smoke artifact across a sequence of hot-path passes:

# each line is a landed tuning packet, oldest first
batch replies on the write side
drain the query buffer once per read batch
direct-write ordinary request/reply traffic
batch client-info snapshots
reuse argv storage instead of reallocating
monotonic timing on the hot path
hold the DB0 lock across safe read batches
cache generated command metadata
skip argv snapshots unless slowlog / AOF / replication need them
skip idle standalone replication propagation
fold handler + metadata lookup into one dispatch table
bucket the dispatch table by command prefix
Benchmark artifacts record the commit, runner parameters, and hardware fingerprint so results from different machines are not silently merged. The latest performance playbook links the release-candidate artifacts used for the public alpha numbers.

What remains

The release-candidate matrix is strong enough for an alpha launch, but it is not the end of performance work. The missing public evidence is broader workload coverage, longer soak, tail-latency under contention, and multi-node/persistence hot-path measurements.

The next architectural milestone is specified in RUNTIME_OWNERSHIP_PLAN.md. The current public claim should stay modest: competitive alpha telemetry on surveyed single-node workloads, not universal speed parity.

Reproduce it

# standard suite: -n 1,000,000 -c 50 -P 100 -d 64
$ bash harness/bench/run.sh

# the profile matrix (the harness optimization target)
$ bash harness/bench/run-profile-matrix.sh

# benchmark the published Docker image without local redis-benchmark
$ IMAGE=ghcr.io/ianm199/valkey-rs:alpha PIPELINE=100 TESTS=get,set,incr,ping_mbulk bash harness/docker/bench.sh

# rebuild the live history dashboard from recorded results
$ python3 harness/bench/history.py
